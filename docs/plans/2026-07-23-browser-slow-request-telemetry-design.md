# Rainbond Browser Slow Request Telemetry Design

## 一、项目背景

### 1.1 项目架构

Rainbond UI is a React 16.8 application. Standard API calls pass through
`src/utils/request.js`, which uses Axios and already reports API errors through
the lightweight Sentry client in `src/sentry.js`. Runtime Sentry configuration
is generated by `rainbond-console` and embedded in the initial HTML response.

### 1.2 现有基础

- `rainbond-ui/src/utils/request.js` is the shared request path for standard UI APIs.
- `rainbond-ui/src/sentry.js` sends Sentry envelopes without the full tracing SDK.
- `rainbond-ui/src/sentryConfig.js` normalizes dynamic URL segments and removes query values.
- `rainbond-console/goodrain_web/sentry_config.py` emits frontend Sentry runtime configuration.
- `rainbond-console/console/utils/offline.py` defines offline mode through
  `DISABLE_DEFAULT_APP_MARKET`.
- `rainbond-console/console/views/sentry_proxy.py` proxies existing browser error envelopes.

### 1.3 核心需求

- Monitor user-visible HTTP latency without tracing every request.
- Decide a 1% sample before timing a request.
- Send a Sentry transaction only when the sampled request exceeds 1000 ms.
- Send slow-request transactions directly from the browser to central Sentry so
  Rainbond Console and Region services receive no telemetry forwarding load.
- Never retry or persist failed telemetry.
- Stop browser Sentry initialization and Console proxy forwarding when
  `DISABLE_DEFAULT_APP_MARKET=true`; keep the Console server SDK configuration
  unchanged.
- Keep telemetry payloads free of tenant, user, application, component ID,
  token, header, body, query, host, and raw URL data. The static component label
  `rainbond-ui` is allowed.

## 二、用户旅程

### 2.1 用户操作流程

- Users configure nothing. Online installations initialize the existing browser
  Sentry client when a DSN is present.
- Before each eligible standard UI request, the browser performs one 1% random
  sampling decision. Unsampled requests retain the original code path.
- Direct `fetch` and streaming request paths are outside the first version.
  Shared-wrapper callers set `skipSlowRequestTelemetry: true` for static,
  streaming, upload, or download traffic that should not be measured; omitted
  or false means eligible for sampling.
- A sampled request is timed. If it completes in 1000 ms or less, no event is built.
- A sampled request over 1000 ms emits a fire-and-forget Sentry transaction.
- Maintainers view grouped slow endpoints in Sentry Performance or Discover.
- Offline installations set `DISABLE_DEFAULT_APP_MARKET=true`; no Sentry client is
  initialized and stale proxy requests are discarded locally with HTTP 204.

### 2.2 页面原型

No Rainbond UI page, dialog, form, or navigation entry is added. Results are
viewed only in the existing external Sentry interface.

### 2.3 外部系统交互

- Online browser to central Sentry envelope endpoint over HTTPS.
- Existing browser error events may continue to use the Console Sentry tunnel.
- Slow-request transactions bypass the Console tunnel.
- Offline mode makes no upstream Sentry request.

## 三、整体架构设计

### 3.1 系统架构图

```text
rainbond-ui request()
  -> eligibility check
  -> 1% random sample
  -> Axios request
  -> duration > 1000 ms
  -> privacy allowlist + page safety cap
  -> sendBeacon or fetch(keepalive)
  -> central Sentry

DISABLE_DEFAULT_APP_MARKET=true
  -> rainbond-console runtime config enabled=false and dsn=""
  -> rainbond-ui skips Sentry initialization
  -> rainbond-console Sentry proxy returns 204 without upstream I/O
  -> rainbond-console server SDK configuration is unchanged
```

### 3.2 核心流程

Sampling happens before timing. Only sampled requests read the timing clock or
allocate a timing context. Timing begins immediately before Axios invocation and
ends at the first Axios settlement, before response processing and error handlers.
Completion is idempotent so errors thrown after an Axios response cannot emit a
second transaction. Cancellation only discards slow telemetry; the existing
Axios catch path, return behavior, and error-event behavior remain unchanged.
Slow transaction sending does not await a response, retry, use local storage, or
retain an in-memory queue. A rolling list of emission-attempt timestamps allows
at most 10 attempts in any 60-second period, with fake-clock boundary tests.

## 四、数据模型设计

### 4.1 新增数据库表

No database table or persistent client storage is added.

### 4.2 数据关系

Each sampled slow request maps to one Sentry transaction event. The transaction
name is `METHOD normalized-route`; request duration is represented by transaction
start and end timestamps. `count()` represents sampled events, not exact traffic.

## 五、API设计

### 5.1 接口列表

No Rainbond business API is added or changed. The existing Sentry envelope ingest
protocol is used with an envelope item type of `transaction`.

### 5.2 请求/响应结构

The browser transaction uses an allowlist:

- transaction name: HTTP method plus normalized path
- operation: `http.client`
- timestamps: request start and end
- tags: component, source, method, status code, sample rate, release
- trace context: generated trace ID, span ID, operation, and status

The `component` tag is the static value `rainbond-ui`, never a Rainbond component
identifier. A request without a response uses the status value `network_error`.
The transaction excludes users, resource identifiers, headers, request/response
bodies, query values, hostnames, and raw URLs. Telemetry responses are ignored.

## 六、核心实现设计

### 6.1 关键逻辑

- Add pure slow-request eligibility, route normalization, sampling, completion,
  rolling-budget, envelope, and event-builder helpers with Node tests.
- Route normalization is fail-closed: parse only API paths, discard malformed or
  encoded-separator paths, retain only allowlisted static API segments, replace
  known resource values and every unknown segment with placeholders, and remove
  the entire query and host. If a safe route cannot be produced, do not sample.
- Keep constants internal: threshold 1000 ms, sample rate 0.01, page cap 10/minute.
- Add `startSlowRequestTracking` and `finishSlowRequestTracking` to `sentry.js`.
- Extend envelope sending to select `event` or `transaction` item types and to
  choose the direct Sentry endpoint for slow transactions.
- Instrument both Axios success and failure paths in `request.js` without changing
  existing return values or error behavior. Cancelled requests are ignored.
- Exclude `/console/sentry`, `/console/posthog`, direct fetch/streaming paths, and
  requests with `skipSlowRequestTelemetry: true`. Upload/download callers using
  the wrapper use this explicit option instead of fuzzy URL matching.
- Reuse `is_offline_mode()` only in frontend Sentry configuration and proxy
  handling. `DISABLE_CLOUD_MARKET` alone does not disable this feature.

### 6.2 复用现有代码

- Reuse existing path parsing primitives, but use a stricter slow-request route
  template helper instead of emitting `getPathPattern()` output directly.
- Reuse the existing browser client DSN parser and fire-and-forget transport.
- Reuse `console.utils.offline.is_offline_mode()` rather than introducing another
  environment variable or offline-mode parser.
- Do not reuse error-event user or Rainbond resource tags for performance events.

## 七、实施计划

### 跨层覆盖检查

- [ ] Go (rainbond): 不涉及 - no server-side request collection.
- [ ] Python (console): 需要 - offline runtime configuration and proxy short circuit.
- [ ] React (rainbond-ui): 需要 - sampling, timing, transaction building, and direct transport.
- [ ] Plugin frontend (enterprise-base): 不涉及.
- [ ] Plugin backend (plugin-template): 不涉及.

### Sprint 1: Offline control

#### Task 1.1: Disable Sentry in offline installations

- 仓库：rainbond-console
- 文件：`goodrain_web/sentry_config.py`, `console/views/sentry_proxy.py`, related tests
- 实现内容：reuse `is_offline_mode()` for frontend runtime configuration and
  return HTTP 204 from the Sentry proxy without upstream I/O. Preserve backend
  SDK configuration.
- 验收标准：offline frontend configuration contains no DSN; proxy mock is never
  called; backend config and `DISABLE_CLOUD_MARKET` behavior remain unchanged.

### Sprint 2: Browser slow-request collection

#### Task 2.1: Add pure sampling and transaction helpers

- 仓库：rainbond-ui
- 文件：`src/sentryConfig.js`, `src/sentryConfig.node.test.js`
- 实现内容：testable constants, fail-closed route templates, exclusions, 1%
  sampling, rolling budget, trace status mapping, envelope selection, and
  privacy-safe transaction construction.
- 验收标准：unit tests cover malformed/absolute/encoded/unknown paths, exact
  recursive payload keys, forbidden Axios data, boundary timing, cancellation,
  budget windows, and transport target/item type selection.

#### Task 2.2: Integrate direct transaction transport and Axios timing

- 仓库：rainbond-ui
- 文件：`src/sentry.js`, `src/utils/request.js`
- 实现内容：pre-request sampling, idempotent completion, 1000 ms filter,
  page-level safety cap, and direct fire-and-forget transaction envelopes.
- 验收标准：automated tests prove unsampled requests do not read the clock,
  success/failure completion emits at most once, direct transactions use
  `transaction` while errors retain `event` plus the Console tunnel, and transport
  performs one fire-and-forget attempt. Production build passes. A live central
  Sentry smoke check is a deployment acceptance step because it requires the
  deployment's public DSN and CORS configuration; it must not use stored secrets
  in repository tests.

#### Task 2.3: Opt transfer callers out before sampling

- 仓库：rainbond-ui
- 文件：`src/services/aiEngine.js`, `src/services/createApp.js`, transfer opt-out test
- 实现内容：set `skipSlowRequestTelemetry: true` on the actual model artifact
  upload and package chunk upload requests. Keep upload initialization,
  completion, status, and cancellation control APIs eligible because they are
  ordinary small API calls. Direct-fetch transfers remain outside the wrapper.
- 验收标准：a source-level Node test proves both transfer call sites contain the
  explicit option, and the production build passes.
